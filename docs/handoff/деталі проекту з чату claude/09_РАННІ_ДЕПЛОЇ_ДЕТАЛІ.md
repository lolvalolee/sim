## ДЕПЛОЙ 25-fix — РЕСТАРТ ГРУПИ + ДАТА (важливо для теми ДАТ!)
Три баги рестарту, усі полагоджені — але дотичні до поточної проблеми дат:
- Баг A (рестарт→500): був `.run(...).catch?.()` — better-sqlite3 СИНХРОННИЙ, .catch
  фіктивний, помилка не ловилась. Тепер wipeStudentSession() з try/catch, кожен DELETE
  перевіряє існування таблиці.
- Баг B (стара дата після рестарту): reset видаляв sessions, але лишав ХВІСТ —
  incidents, order_events, resume_points, application_followups, confirmations,
  applications. Старі інциденти з минулим scheduled_at спрацьовували одразу →
  "час пройшов". ТЕПЕР чистяться ВСІ дочірні таблиці (по session_id) + резюме (student_id).
- Баг C (синхронізація дати): рестарт пише дату в groups.start_date + started_at.
  Session при створенні бере start_date з групи → всі студенти стартують з однієї
  дати = День 1, прискорений час іде від неї.
- Дата старту: кнопка рестарту відкриває модалку — чекбокс "Поточна дата" (сьогодні)
  або вибір з календаря. Формат ДД.ММ.РРРР.
- Баг D (autofill): поле пошуку перевізників мало браузерний autofill email входу.
  Фікс: readonly+зняття на focus, data-lpignore, data-1p-ignore.

⚠️ ВАЖЛИВО ДЛЯ ТЕМИ ДАТ: ці фікси показують, що дати/сим-час крихкі. Корінь часто —
старі дані в дочірніх таблицях із минулим scheduled_at. При роботі над "дати
відскакують" перевірити: чи всі таблиці чистяться при рестарті, чи інциденти
прив'язані до сим-часу (target_sim_hour_abs), а не реального scheduled_at.
