# ТОРГ — ДЕТЕРМІНОВАНА МЕХАНІКА (Варіант B)

## ГОЛОВНИЙ ПРИНЦИП
Ціну рахує КОД, не AI. AI отримує готову цифру + дію і лише формулює фразу.
Це доведено необхідним: промпт НЕ тримає межі — AI пробивався в обидва боки
(перевізник скидав 3500→1900, замовник погоджувався на +700 за "послуги").

## КОРИДОРИ (від freight_ref конкретного рейсу з файлу Симулятор)
- ПЕРЕВІЗНИК (студент платить йому, хоче більше): ref−300 … ref+100
- ЗАМОВНИК (платить студенту, хоче дешевше): ref … ref+300
- МАКС КОМІСІЯ = (ref+300) − (ref−300) = 600 EUR. Пробити неможливо (рахує код).

## КРОК ПОСТУПКИ
- 50 або 100 EUR, рандомно, КРАТНО 50. Дрібні (10-40) НЕ практикуються.
- Залежить від характеру (_negStep): tough → 50 (впертий), local → 100 (поступливий).
- Перша ставка перевізника варіюється (не всі однаково): ref+50…+100 за характером.

## ФУНКЦІЇ (simulator.html ~рядок 4248, клієнтський JS)
```javascript
_negStep(pers)                              // 50 або 100 за характером
carrierFirstOffer(ref, pers)                // перша ставка перевізника = ref+50..100
carrierNextOffer(ref, current, studentOffer, pers)
  // → {offer, action}; action: 'agree'|'concede'|'hold'
  // studentOffer >= current → agree (вигідно перевізнику)
  // studentOffer >= floor(ref-300) → agree на запропоноване
  // нижче → поступається кроком, але hold на floor (ref-300) — НІКОЛИ нижче
clientFirstOffer(ref)                       // = ref (нижня межа що платить)
clientNextOffer(ref, current, studentAsk)
  // studentAsk <= current → agree
  // studentAsk <= ceil(ref+300) → піднімається кроком до запиту
  // вище ceil → hold на ceil (ref+300) — НІКОЛИ вище
extractOfferNum(text)                       // витягує число (EUR) з тексту студента
```

Дзеркало цих функцій є і в server/utils/negotiation.js (СТВОРЕНО, але student.js
його НЕ реквайрить — торг рахується на клієнті, серверний модуль для майбутнього/тестів).

## ЯК ВБУДОВАНО (sendChatMsg перевізник ~3470, sendEmailReply замовник ~2800)
1. Визначається ordCtx (рейс) → freightRef рейсу.
2. Якщо carrierNegoOffer == null → carrierFirstOffer (стартова).
3. extractOfferNum з останнього повідомлення студента → studentOffer.
4. carrierNextOffer(...) → {offer, action}.
5. Оновлюється стан (ordCtx.carrierNegoOffer = offer), persist через PATCH /orders/:id/nego.
6. negoDirective з готовою цифрою додається в історію перед callClaude.
7. AI формулює фразу з цією цифрою (не вигадує своєї).

## ПРИКЛАД (ref=1700)
Перевізник: старт 1800 (tough). Студент 1500→agree; 1400→agree (межа); 1200→hold 1400; 900→hold 1400.
Замовник: старт 1700. Студент просить 1800→agree; 1950→піднявся 1900; 2000→agree; 2500→hold 2000.
Комісія завжди ≤ 600.

## ЦІНА ЗА ТОННУ (цистерна) — ВІДОМИЙ РИЗИК
Якщо для рейсу ставка у файлі вказана ЗА ТОННУ (а не загальна сума) — freight_ref
треба перерахувати в загальну (ставка × тонни), інакше коридор рахується від
ставки-за-тонну і виходить абсурд. ПЕРЕВІРИТИ при додаванні таких рейсів.
(На проді всі 120 freight_ref у діапазоні 750..7500 — схоже, вже загальні суми.)

## АНТИШАХРАЙСТВО (ПОБАЖАННЯ ВЛАСНИКА, ще не реалізовано)
Якщо студент: сильно випрошує / маніпулює "додатковими послугами" (страхування,
кріплення трактора, супровід через митницю — це реальний кейс №8 Правки_8, де
замовник погоджувався на +700) / тисне нечесно — це спроба ШАХРАЙСТВА. Тоді
перевізник АБО замовник (саме той, хто побачив маніпуляцію) каже "вибач, друже,
більше не цікаво" і ВЗАГАЛІ відмовляється від угоди. Детермінований запобіжник:
аргументи про "послуги" НЕ зрушують стелю ref+300 (для замовника) / floor ref-300
(для перевізника). Код ігнорує текстові аргументи, тримає цифру.
