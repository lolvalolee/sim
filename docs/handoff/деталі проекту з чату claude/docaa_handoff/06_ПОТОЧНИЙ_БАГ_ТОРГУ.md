## ДІАГНОЗ — ЧОМУ ТОРГ ВСЕ ОДНО ПЛИВЕ

Торг рахується лише якщо `fref > 0` (simulator.html ~рядок 3474):
```javascript
const fref = ordCtx?.freightRef || ordCtx?.clientFreight || 0;
if (fref > 0 && ordCtx) { ... carrierNextOffer ... }
```
Якщо fref == 0 → B-торг ВИМИКАЄТЬСЯ → AI торгується вільно → пливе.

fref = 0 коли ordCtx не визначено (рейс не закріплено). А це буває коли
matchOrderByRoute НЕ знайшов рейс за маршрутом у повідомленні студента.

### ПЕРЕВІРЕНО:
- Дані letters↔letters_v2: 120/120/120 linked (ЦІЛІ). freight_ref у всіх 750..7500. НЕ причина.
- Отже причина — matchOrderByRoute не матчить АБО серверний підбір біржі.

## КОРІНЬ №1 (зі скріна Правки_8 image1) — БІРЖА, ФІЛЬТР НАПРЯМКУ

Скрін: перевізник Хоменко (напрямки UA, FI) ВІДГУКНУВСЯ з біржі на вантаж
Айзенах→Житомир (Айзенах = DE, тобто рейс DE→UA), якого НЕ возить. Торгувався,
потім сам сказав "це не наш напрямок, ми UA-FI, Німеччину не возимо".

Корінь у student.js exchange/post (~рядок 2413-2421), фільтр matching:
```javascript
const dirOk = dirsNeeded.length === 0 || dirsNeeded.some(d => cdirs.includes(d));
```
ЛАЗІВКА: якщо dirsNeeded ПОРОЖНІЙ (лист не знайдено за letter_id І за маршрутом),
фільтр напрямку ПРОПУСКАЄТЬСЯ → беруться перевізники БУДЬ-ЯКИХ напрямків →
UA-FI перевізник проходить на DE→UA вантаж.

Айзенах→Житомир — біржовий пост. Якщо letter_id не переданий і route-матчинг
(рядки 2386-2402, грубий `every part includes`) не знайшов "Айзенах" у адресах
листів → dirsNeeded=[] → фільтр напрямку off → невідповідний перевізник.
А також baseFreight=0 при порожньому → start_price=0 → ціна пливе.

## ЩО РОБИТИ (узгоджено, ще НЕ закодовано)

Власник підтвердив флоу: "студент виставляє вантаж на біржу — обирає рейс з листа".
Тому ПИТАННЯ перед кодом (НЕ з'ясоване остаточно):
**Коли студент постить на біржу — є letter_id (вибрав зі списку листів)
чи вписує маршрут руками?**
- Якщо letter_id Є → передавати його на сервер exchange/post → dirsNeeded і
  baseFreight завжди точні. Найнадійніше. (ТРЕБА перевірити чи фронт його шле.)
- Якщо маршрут руками → треба надійніший матчинг тексту до листа (нормалізація міст).

### ФІКС (детермінований, коли визначимось із letter_id):
1. ЗАВЖДИ передавати letter_id з біржі на exchange/post (якщо студент постить з листа).
2. ПРИБРАТИ лазівку `dirsNeeded.length === 0 ||` — якщо напрямок невідомий,
   НЕ підбирати перевізників навмання (краще 0 відгуків, ніж невідповідні).
3. matchOrderByRoute (для чату) — покращити розпізнавання міст (Айзенах/Eisenach,
   складені назви типу "Вінер Нойштадт", латиниця/кирилиця).

## КОРІНЬ №2 (Правки_8 п.2, 4, 16) — МАРШРУТ/ДВА РЕЙСИ

## matchOrderByRoute — ПОТОЧНИЙ АЛГОРИТМ (simulator.html ~4291)
Витягує слова-міста з тексту студента, шукає рейс де loadAddress/unloadAddress/dirs
містить збіги (бонус +2 якщо і відправлення, і призначення). Потрібен ≥1 збіг.
СЛАБКЕ МІСЦЕ: підрядковий includes — не нормалізує написання, не знає синонімів
міст, чутливий до того як студент написав ("Айзенах" vs "Eisenach" vs з індексом).
